# ログを観測の単一ソースにする

## この記事で扱うこと

[CodeRouter](https://github.com/zephel01/CodeRouter) は Claude Code / codex / gemini-cli の手前に立つ小さな LLM ルーターです。v1.5.0 で観測まわりを 6 サブリリース (A-F) に切って一気に積みました。本稿では、その中で**繰り返し効いた 4 つの設計判断**を書きます。

1. **`MetricsCollector` を `logging.Handler` のサブクラスにする** — 既存ログを唯一のソースに置き、計測コードを業務ロジックに 1 行も挿さない
2. **集計は UTC、表示だけタイムゾーン変換** — wire は不変、表示層に責務を閉じ込める
3. **CLI TUI と HTML ダッシュボードを `/metrics.json` の 2 コンシューマにする** — pure data + render layer を共有、driver だけ別
4. **Prometheus exposition は手書き 30 行で十分** — `prometheus_client` を入れない選択

依存ランタイムは v1.0 系から据え置きの 5 個、Python 3.12+、`asyncio` + `httpx` + `fastapi`、テスト 457 → 527 (+70)。`curses` / `urllib` / `zoneinfo` は全て stdlib、tailwind は CDN 1 ファイル、Prometheus 形式は自前文字列生成です。

![CodeRouter dashboard — 5 つのパネル (Providers / Fallback & Gates / Requests/min sparkline / Recent Events / Usage Mix)](https://raw.githubusercontent.com/zephel01/CodeRouter/v1.5.0/docs/assets/dashboard-demo.png)

## 設計判断 1: `MetricsCollector` を `logging.Handler` にする

CodeRouter のログは v0.3 以降、構造化された 1 行 JSON で固定されています:

```python
logger.info("try-provider", extra={"provider": "ollama", "profile": "code"})
logger.info("provider-ok",  extra={"provider": "ollama", "stream": True})
logger.warning("provider-failed", extra={"provider": "ollama", "status": 503,
                                          "retryable": True, "error": "..."})
```

事件名 (`record.msg`) と context (`record.__dict__`) で全て表現できる、というのが v0.3 で決めた不変項です。v1.5-A の `MetricsCollector` はこれを直接利用します。

```python
# coderouter/metrics/collector.py
class MetricsCollector(logging.Handler):
    def emit(self, record: logging.LogRecord) -> None:
        try:
            self._dispatch(record)
        except Exception:                # 計測でログ呼び出しを爆発させない
            self.handleError(record)

    def _dispatch(self, record: logging.LogRecord) -> None:
        event = record.msg
        if not isinstance(event, str):
            return
        extras = record.__dict__
        with self._lock:
            if event == "try-provider":
                self._requests_total += 1
                self._provider_attempts[_str(extras.get("provider"))] += 1
                self._push_recent(event, extras, record)
            elif event == "provider-ok":
                # ...
            elif event == "provider-failed":
                # ...
```

`addHandler()` するだけで自動的に発火します。**業務ロジックには計測コードを 1 行も足していません**。「ログ to メトリクス」のパイプラインで一番起きがちな「メトリクス専用の `metrics.inc(...)` 呼び出しが各所に散る → ログとメトリクスが二重実装で乖離する」という failure mode を構造的に避ける配置です。

3 つの帰結があります:

- **追加コスト**は collector 側のディスパッチテーブル 1 行のみ。新しいイベントを増やすコードを書く人は、ロジックを書きながらログを書けばよく、メトリクスのことを意識しなくていい。
- **未知イベントは黙って無視**。dispatch テーブルに無い `event` 名は素通り。ログ側で新イベントを足したときに collector が壊れない (forward-compatible)。
- **失敗の隔離**: `emit()` 内で例外が出ても `handleError()` で吸収。計測の不具合が業務ログをクラッシュさせるのは絶対に避ける。

スレッドセーフは `threading.RLock`。`logging.Handler.emit` は logging モジュールが `handler.lock` を取って呼ぶので二重ロックですが、`snapshot()` を別スレッド (FastAPI のイベントループ) から呼ぶときに**一貫したビュー**を返す必要があり、自前 lock は外せません。

## 設計判断 2: 集計は UTC、表示だけタイムゾーン変換

v1.5-E で `display_timezone: "Asia/Tokyo"` を `providers.yaml` の top-level に足しました。ここで一番強く守った原則は「**集計層に timezone を入れない**」です。

```python
# collector.py — 内部時刻は常に UTC
def _utc_now_iso() -> str:
    return datetime.now(UTC).strftime("%Y-%m-%dT%H:%M:%S")
```

`/metrics.json` のレスポンスも UTC ISO のまま:

```json
{
  "uptime_s": 1234.5,
  "started_at": "2026-04-22T01:23:45",
  "config": {"display_timezone": "Asia/Tokyo"},
  "recent": [
    {"ts": "2026-04-22T01:24:30", "event": "provider-ok", "provider": "ollama"}
  ]
}
```

変換は表示層 2 種で:

- **CLI TUI**: snapshot 1 回につき `zoneinfo.ZoneInfo(name)` を 1 度だけ生成 (stdlib 側で同名 zone はキャッシュされる)、`datetime.fromisoformat(s).replace(tzinfo=UTC).astimezone(zone)` で各行を変換
- **HTML dashboard**: `Intl.DateTimeFormat(displayTz, ...)` をブラウザネイティブで実行、`/metrics.json` の `config.display_timezone` を JS 側に伝搬

「集計は UTC、表示は local」を守ると以下の bug class が消えます:

- DST 切り替え時に counter が二重カウント / スキップされる
- 異なる zone のオペレータが同じ JSONL を読んだとき行の順序が混乱する
- Prometheus exposition の timestamp に tz suffix が混じって `promtool` が rejection する

逆に「やってない」のが、**JSONL mirror に local zone を書く** 選択。`$CODEROUTER_EVENTS_PATH` に書き出すファイルは UTC のまま固定、後段の log 解析パイプライン側で必要なら変換してください、という分担にしています。

## 設計判断 3: pure data+render layer を 2 コンシューマで共有する

v1.5-C (curses TUI) と v1.5-D (HTML dashboard) は、見た目は別物ですが**同じ意味論の 5 パネル**を表示します:

| パネル | 答える問い | データ源 |
|--------|-----------|---------|
| Providers | 誰がいま応えているか | `provider_attempts` + `provider_outcomes` + `last_error` |
| Fallback & Gates | 直近で切り替わったか | `chain_paid_gate_blocked_total` + `capability_degraded` |
| Requests/min sparkline | 直近数分の流量 | client-side ring (60 sec rolling) |
| Recent Events | 直近で何が起きたか | `recent` deque (size 256) |
| Usage Mix | local / free / paid の比率 | `provider_outcomes` 集計 |

これを 2 度実装するのを避けるため、**driver と pure layer を分離**しました:

```python
# coderouter/cli_stats.py
@dataclass(frozen=True)
class ProviderRow:
    name: str
    attempts: int
    ok: int
    failed: int
    success_rate: float
    health: str  # "green" | "yellow" | "red" | "gray"
    last_error: str | None

def build_provider_rows(snapshot: dict[str, Any]) -> list[ProviderRow]:
    """Pure transform: snapshot dict → list[ProviderRow]. No I/O."""
    ...

def run_once(snapshot: dict[str, Any]) -> str:
    """Plain text driver — for non-tty, --once mode, demo banner."""
    ...

def run_tui(...) -> None:
    """curses driver — wraps build_* + render."""
    ...
```

unit test は **pure layer (`build_*`) と format (`format_text`) だけ**を叩きます。curses も urllib も触らないので、テストはミリ秒で 39 本 (`tests/test_cli_stats.py`) 走ります。

HTML 側 (v1.5-D) は同じ意味論を JS で再実装していますが、データ源は `/metrics.json` の同じ snapshot dict なので、CLI と HTML が表示する数字は**バイト一致**で一致します。「TUI で見えていたものが dashboard では違って見える」という gnarly な乖離 bug が原理的に起きない配置です。

副次効果として `--once` モードで `coderouter stats --once` を `scripts/demo_traffic.sh` の banner / CI ログに埋め込めます。同じ render layer が「ライブ更新」と「単発スナップショット」で動くので、demo 用に別形式を起こす必要がありませんでした。

## 設計判断 4: Prometheus は手書き 30 行で十分

v1.5-B で `GET /metrics` を Prometheus text exposition v0.0.4 で出します。`prometheus_client` を入れず、`coderouter/metrics/prometheus.py` の `format_prometheus(snapshot)` で文字列を生成しています。

```python
# coderouter/metrics/prometheus.py
_PREFIX = "coderouter_"

def format_prometheus(snapshot: dict[str, Any]) -> str:
    lines: list[str] = []
    counters = snapshot.get("counters", {})

    lines.extend(_gauge(
        name="uptime_seconds",
        help_text="Seconds since the CodeRouter process started.",
        value=snapshot.get("uptime_s", 0.0),
    ))
    lines.extend(_counter(
        name="requests_total",
        help_text="Total requests dispatched to the fallback engine.",
        samples=[((), counters.get("requests_total", 0))],
    ))
    # ... 全部で 211 LOC
    return "\n".join(lines) + "\n"
```

選択理由は plan.md §12.3.4 で言語化してありますが、要約すると:

- **format が安定** — Prometheus exposition v0.0.4 は 2017 から仕様変更なし
- **二重簿記を避ける** — `prometheus_client` を入れると collector が `prometheus_client.Counter` を inc しに行く必要が出てきて、log-tap のシングルソース原則が壊れる
- **E2E 検証は `promtool check metrics`** で外部担保。SDK が無くても spec compliance は確認できる
- **依存予算** — 5 個ルールに対して `prometheus_client` は 100kB+ で重い

ラベルなしの scalar counter / gauge しか使っていないので、ラベル escape も必要最小限 (provider 名は user 設定経由なので `\` `"` `\n` のみ escape) です。

JSONL mirror (`$CODEROUTER_EVENTS_PATH`) も同じ思想で、`logging.FileHandler` + 既存の `JsonLineFormatter` の組み合わせで実装。`logrotate` は外部任せ。`RotatingFileHandler` を入れない判断は plan.md §12.3.3 に書いてあります。

## まとめ

v1.5.0 の観測まわりは、**単一の構造化ログをソースにして、そこから配信先を分岐させる**という古典的な設計をきっちり守ることで、6 サブリリースを横断して整合性を保てました。

- 計測の追加コストはディスパッチテーブル 1 行
- 集計は UTC、表示だけ tz 変換 (DST バグを構造的に消す)
- TUI と dashboard は同じ snapshot を見る (「見え方が違う」を起きない設計に)
- Prometheus は 30 行で書いて 5-dep 予算を守る

ソースは [zephel01/CodeRouter](https://github.com/zephel01/CodeRouter)、v1.5.0 タグでチェックアウトできます。`coderouter serve` 起動後に `localhost:4000/dashboard` / `coderouter stats` / `curl localhost:4000/metrics` をそれぞれ叩けば、上で説明した 4 つの設計判断が同じ snapshot を経由して 4 通りに見える様子が確認できます。
