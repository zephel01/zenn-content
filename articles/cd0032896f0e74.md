# CodeRouter の内側

## この記事で扱うこと

[CodeRouter](https://github.com/zephel01/CodeRouter) は、Claude Code や gemini-cli のような LLM エージェントの手前に立つ小さなルーターです。ユーザーから見れば「ローカル Ollama にフォールバックつきで繋ぎ込めて無料で動く便利ツール」ですが、**実装目線で見ると、SSE ストリーミングプロトコルの 2 方言間を壊れずに翻訳しきる**という、地味にしんどい問題の塊です。

この記事では、CodeRouter の v1.0 時点のコアに据わっている 4 つの設計判断を取り上げます:

1. **Anthropic ⇄ OpenAI の双方向ワイア変換** — ingress と provider kind の直交化
2. **Mid-stream フォールバックガード** — 「最初のバイトを出した後は切替禁止」という不変項
3. **ステートフル SSE フィルタチェーン** — `<think>` と stop マーカーを、チャンク境界をまたいで安定に剥がす
4. **`doctor` の実地プローブ設計** — ヒューリスティクス依存からの脱却

依存はランタイムで 5 個、Python 3.12+、pure 非同期 (`asyncio` + `httpx` + `fastapi`)、テスト 453 本で CI 緑。

## 全体構成

```
┌────────── Client ──────────┐
│ Claude Code (Anthropic)    │
│ gemini-cli / codex (OAI)   │
└────────────┬───────────────┘
             │
      ┌──────▼────────────┐
      │      Ingress      │  — FastAPI
      │  /v1/messages     │  (Anthropic 受け)
      │  /v1/chat/comps   │  (OpenAI 受け)
      └──────┬────────────┘
             │  ChatRequest | AnthropicRequest (pydantic)
             ▼
      ┌───────────────────┐
      │ Routing / Profile │  — profile = body > header > mode alias > default
      │   capability gate │  — thinking / prompt-cache / reasoning
      └──────┬────────────┘
             │  Sequence[ProviderCallTask]
             ▼
      ┌───────────────────┐
      │  Fallback Engine  │  — 逐次実行、mid-stream ガード付き
      └──┬────────────────┘
         │
   ┌─────▼────────┐  ┌─────▼────────┐
   │ openai_compat│  │  anthropic   │    (Adapter — kind 別)
   │   adapter    │  │   adapter    │
   └─────┬────────┘  └─────┬────────┘
         │                 │
   ┌─────▼────────┐  ┌─────▼────────┐
   │ httpx to ... │  │ httpx to     │
   │ Ollama /     │  │ api.anthropic│
   │ OpenRouter / │  │ .com         │
   │ Together …   │  │              │
   └──────────────┘  └──────────────┘
```

全部の流れが非同期。`FastAPI` の `StreamingResponse` の `AsyncIterator[bytes]` は、最終的に上流の `httpx.AsyncClient.stream(...)` の `aiter_bytes()` まで **生成パイプライン 1 本**で繋がります（途中で 3 回くらい変換がかかっていても、ジェネレータを `async for` で消費する形のまま）。

## 設計判断 1: Ingress と Provider kind を直交させる

一番大事な設計判断です。多くのルーターは「入り口 = 出口の形式」を暗黙に仮定しています (例: OpenAI で受けたら OpenAI に出す)。CodeRouter は **2 × 2 の直交マトリクス**で全組み合わせを動かします:

|                       | provider `openai_compat` | provider `anthropic` |
|-----------------------|--------------------------|----------------------|
| ingress `/v1/messages` (Anthropic) | Anthropic → OpenAI 変換 → 戻し変換 | パススルー（ワイア無変換） |
| ingress `/v1/chat/completions` (OpenAI) | パススルー | OpenAI → Anthropic 変換 → 戻し変換 |

この 4 セルが全部動く、というのが CodeRouter の地味な強みです。特に **左下** ("OpenAI で受けて Anthropic を叩く") のセルは v0.4-A で追加したもので、実用上は:

- codex (OpenAI 形式) でもフォールバック先に有料の Anthropic Claude を置ける
- gemini-cli (OpenAI 互換モード) から Claude にオーバーフローできる
- Claude Code のユーザが「クライアントを OpenAI SDK に変えても設定は同じ」の選択肢を持てる

という副次効果があります。

### 形式の違いを `content` ブロック構造が吸収する

Anthropic 形式の肝は `messages[].content` が**配列**で、各要素が `{type: "text" | "tool_use" | "tool_result" | "image"}` のバリアントであることです。OpenAI Chat Completions は `messages[].content` が**文字列**（または `gpt-4-vision` 向けの限定的なブロック配列）で、tool 呼び出しは別フィールド `tool_calls` に出ます。

変換は双方向で:

**Anthropic → OpenAI**

- `content[]` をスキャンして `type: "text"` を string concat → `message.content`
- `type: "tool_use"` を `message.tool_calls[]` に移動、`input` を `JSON.stringify` して `function.arguments` に
- `role: "user"` のターンに `type: "tool_result"` ブロックが混ざっている場合、それらを `role: "tool"` の別メッセージに**ばらす**（OpenAI はロールを分ける）
- system メッセージは `messages` 配列の外の `system` フィールドから取り出して `messages[0]` に持ち上げる

**OpenAI → Anthropic** (逆翻訳、v0.4-A で追加)

- `message.content` (string) を `content: [{type: "text", text: ...}]` にラップ
- `message.tool_calls[]` を `type: "tool_use"` ブロックに変換、`function.arguments` を `JSON.parse` して `input` に
- `role: "tool"` の連続を 1 つの `role: "user"` + `content: [{type: "tool_result", ...}, ...]` に**束ねる**（Anthropic は束ねる）
- 先頭の system を `messages` 外の `system` フィールドに持ち上げる

この「ばらす / 束ねる」の非対称性が、素朴な wire 変換で一番ハマるところです。pydantic モデル (`AnthropicMessage`, `AnthropicContentBlock`, `ChatRequest`) をしっかり定義しておいて、変換関数は**型で不変項を表明する**（union 型の網羅性を `mypy --strict` で担保）スタイルを徹底しました。

### Tool-call リペア: 「形式違反を許容するのは変換の外」

ここが CodeRouter の古株機能 (v0.3-A) で、かつ**一番他のルーターと差別化される**ところ。

問題: `qwen2.5-coder:14b` や `deepseek-coder-v2` のような小型ローカルモデルは、`tool_calls` フィールドを使わずに `content` の中に `{"name": "Read", "arguments": {"file_path": "..."}}` を**生テキストで吐く**。

素朴な実装: 変換層で `content` から JSON を regex で掘って、見つけたら `tool_calls` に入れる。

これは**壊れます**。理由は:

- 本物の「JSON の話をする会話」（例: "ツール呼び出しは `{\"name\":\"Read\"...}` のような形で..." と説明するチュートリアル応答）が誤誘拐される
- 部分的な JSON（`{"name": "Read"`で切れているような）が壊れてパースエラー
- tool name が未宣言の造語（モデルの hallucination）を `tool_calls` に詰めて Claude Code が confused

CodeRouter のリペアは:

1. **Balanced brace scanner** で `{...}` を検出（ネストを数えて最外を抽出）
2. パース成功したものの中から `name` がリクエストに含まれる **allow-list 済み tool 名** と一致するものだけを採用
3. ストリーミング経路では「部分 JSON」を保留して、次チャンクと連結してから再試行

これをリペア層（変換層とは別）に分離したことで、**変換層は「壊れていない入力」前提で書ける**ようになりました。責務分離の教科書例。

## 設計判断 2: Mid-stream フォールバックガード

フォールバックエンジンの不変項:

> **最初のコンテンツバイトをクライアントに送出した後、プロバイダが失敗したら、以降のプロバイダに切り替えてはならない。クライアントには `event: error` を 1 本流して終わる。**

これが無いと、典型的なバグとして:

```
Provider A → [SSE 開始] → "Pythonで" → [接続断] →
Provider B → [別のリクエストで] → "Rubyなら" → [Bが成功] →
クライアントが受け取る: "PythonでRubyなら..."
```

のような**フランケン応答**が発生します。tool 呼び出しの途中でこれが起きると、`tool_use` ブロックが A のものと B のものを混ぜてクライアントに届き、`input` が壊れます。Claude Code は JSON パースに失敗して「この会話が壊れました」と表示するだけで、**なぜ**壊れたかは分かりません。

CodeRouter の実装 (`ProviderRunner`):

```python
async def run_with_fallback(chain: list[ProviderCallTask]) -> AsyncIterator[bytes]:
    first_byte_sent = False
    for task in chain:
        try:
            async for chunk in task.run():
                if not first_byte_sent and _is_content_chunk(chunk):
                    first_byte_sent = True
                yield chunk
            return  # 正常完走
        except MidStreamError:
            # first_byte_sent == True で上流が死んだ
            yield _synthesize_error_event(reason="upstream-died-midstream")
            return
        except ProviderError as e:
            if first_byte_sent:
                # ここに来たら設計違反 — MidStreamError に変換して escape
                yield _synthesize_error_event(reason="invariant-violation", detail=str(e))
                return
            # まだ 1 バイトも出てない → 次のプロバイダへ
            continue
    yield _synthesize_error_event(reason="all-providers-failed")
```

`_is_content_chunk` の判定は wire 形式ごとに違います:

- OpenAI SSE: `data:` に `choices[0].delta.content` か `delta.tool_calls[]` が入っているか
- Anthropic SSE: `event: content_block_delta` が来たか

`message_start` や `content_block_start`（メタデータだけの開始イベント）は**含みません**。これが肝で、もし `content_block_start` で "first byte" と判定してしまうと、「メタ情報だけ出して死んだ」プロバイダの後のフォールバックも禁止されて、可用性が落ちます。

**この不変項のテスト**は 23 本書いてあります。`tests/test_mid_stream_guard.py` で、人工的に「N バイト目で切断」を注入して、fallback 挙動が N < first_content / N ≥ first_content で正しく分岐するか検証しています。

## 設計判断 3: ステートフル SSE フィルタチェーン

v1.0-A で追加した output_filters 機構。設定:

```yaml
providers:
  openrouter-gpt-oss:
    kind: openai_compat
    model: openai/gpt-oss-120b:free
    output_filters: [strip_thinking, strip_stop_markers]
```

こうすると、このプロバイダから流れてくる応答 (ストリーミング / 非ストリーミング両方) から `<think>...</think>` と 6 種の stop マーカーが剥がされます。

### ストリーミングの難しさ

非ストリーミング応答で `<think>` を剥がすのは `re.sub(r"<think>.*?</think>", "", content, flags=re.DOTALL)` で終わり。

ストリーミングはそうはいきません。SSE チャンクは**意図しない境界で分割**されます:

```
チャンク 1: "<think>Let me think about this</thi"
チャンク 2: "nk>The answer is 42"
```

`</thi` の時点で「これは閉じタグの途中かも」と判断して**保留**しないといけません。かと言って、ユーザのプロンプトに `<thi` みたいな文字列が正当に含まれる応答もあり得るので、永久に保留するわけにもいきません。

CodeRouter のフィルタは**ステートマシン**で動きます:

```python
class StripThinkingFilter:
    def __init__(self):
        self.state = OUTSIDE
        self.buffer = ""  # 保留バッファ

    def process(self, chunk: str) -> str:
        self.buffer += chunk
        out = []
        while self.buffer:
            if self.state == OUTSIDE:
                idx = self.buffer.find("<think>")
                if idx == -1:
                    # "<thi" のような部分タグが末尾にあるかチェック
                    safe = _longest_safe_prefix(self.buffer, "<think>")
                    out.append(self.buffer[:safe])
                    self.buffer = self.buffer[safe:]
                    break
                out.append(self.buffer[:idx])
                self.buffer = self.buffer[idx + 7:]
                self.state = INSIDE
            else:  # INSIDE
                idx = self.buffer.find("</think>")
                if idx == -1:
                    # INSIDE では全て捨てる（部分閉じタグの保留は閉じ開始の 8 文字分のみ）
                    safe = len(self.buffer) - 8
                    self.buffer = self.buffer[max(safe, 0):]
                    break
                self.buffer = self.buffer[idx + 8:]
                self.state = OUTSIDE
        return "".join(out)

    def flush(self) -> str:
        # EOF 時の後始末
        if self.state == OUTSIDE:
            return self.buffer  # 保留分はそのまま出す
        # INSIDE のまま EOF → 応答全体を "<think>" 後で切ったことになる
        return ""
```

EOF 処理が非自明で、**OpenAI-compat 経路** では合成 `delta` チャンクで `flush()` の結果を出し、**Anthropic 経路** では合成 `content_block_delta` で出します。Anthropic 経路の方が実は楽で、Anthropic の `content_block_*` はテキストブロックの始端・終端が明示的なので、「ブロック終端の直前に `flush()` を差し込む」という素直な形で書けます。OpenAI は `[DONE]` を送る直前にフィルタの残渣が残っていないか確認しないといけない。

フィルタチェーンは`list[OutputFilter]` の順序通り適用。`strip_thinking` の後に `strip_stop_markers` を置く設計にしてあって、これは「`<think>` の中に `<|eot_id|>` が混ざっている」ケースでの二重剥がしを避けるため（`<think>` ブロックごと捨てるなら中の stop マーカーは気にしなくていい）。

### 未知フィルタ名の fast-fail

細かいが重要: `output_filters: [strip_thiking]`（タイポ）を YAML に書いた場合、**config ロード時に fast-fail** します。初回リクエスト時まで遅延させると、運用中に「なぜかフィルタが効いていない」事態が起きます。`FilterRegistry` に `"strip_thiking"` が無ければ即 `ConfigError` を投げる。登録済みフィルタ名のリストを併記。

## 設計判断 4: `doctor` の実地プローブ

v0.7-B で追加、v1.0-B / v1.0-C で拡張された `coderouter doctor --check-model` は、**ヒューリスティクス依存からの脱却**というテーマの結晶です。

それ以前 (v0.5-C まで) は、実運用中のログから「`capability-degraded` が出た」「tool_use リペアが頻発している」といった**副次的シグナル**で capability を推定していました。これは運用中にリトライが発生するまで判定できない、という致命的な欠点があります。

v0.7-B 以降の doctor は:

- **実地プローブ** (httpx 直叩き) で構造化された刺激を送って、生レスポンスで観測
- 判定表: 各プローブは `OK / NEEDS_TUNING / SKIP / FAIL` の 4 値、全体の exit code は 0 / 1 / 2
- NEEDS_TUNING 時には**コピペ可能な YAML パッチ**を標準出力に

別記事 [zenn-01](./zenn-01-why-same-model-behaves-differently.md) でプローブの因果分離を扱ったので、ここではプローブの**パイプライン構造**に絞ります。

### Pipeline as a list of probes

```python
@dataclass
class ProbeResult:
    name: str
    verdict: Literal["ok", "needs_tuning", "skip", "fail"]
    detail: str
    patch: Optional[YamlPatch] = None

class Probe(Protocol):
    async def run(self, ctx: ProbeContext) -> ProbeResult: ...

PROBES: list[Probe] = [
    AuthProbe(),
    NumCtxProbe(),
    ToolsProbe(),
    StreamingProbe(),
    ReasoningLeakProbe(),
    AnthropicThinkingProbe(),
]
```

順序は [zenn-01](./zenn-01-why-same-model-behaves-differently.md) で解説した通り。前段が FAIL したら後段を `skip` にする:

```python
async def run_doctor(ctx: ProbeContext) -> list[ProbeResult]:
    results: list[ProbeResult] = []
    short_circuit = False
    for probe in PROBES:
        if short_circuit:
            results.append(ProbeResult(probe.name, "skip", "skipped due to earlier failure"))
            continue
        r = await probe.run(ctx)
        results.append(r)
        if r.verdict == "fail" and probe.name in {"auth"}:  # auth 失敗のみ短絡
            short_circuit = True
    return results
```

`auth` 失敗だけを短絡条件にしているのは意図的。他のプローブは失敗しても次に進みます（例: `num_ctx` NEEDS_TUNING のまま `tools` プローブを走らせて、`tools` の結果が `num_ctx` の影響かどうかをレポートに混ぜて出す）。

### Exit code のセマンティクス

```
0 — 全プローブが OK または情報的 OK
2 — NEEDS_TUNING が 1 つ以上（パッチ適用で直る見込み）
1 — UNREACHABLE（auth 失敗、到達不能、未知プロバイダ）
```

CI からは `|| true` せずにそのまま exit code で制御できる設計。

## テストの積層

v1.0 時点で 453 テスト。階層的に:

- **ユニット**: wire 変換関数（Anthropic → OpenAI と逆）、balanced-brace スキャナ、ステートマシンフィルタ、pydantic スキーマバリデーション
- **統合**: FastAPI TestClient で ingress を叩き、モック provider で fallback 挙動を検証
- **契約 (contract)**: 実上流 API 形状のスナップショット（`tests/fixtures/`）に対する逆向き検証
- **プローブ**: `doctor` の 6 プローブを人工レスポンスで叩いて判定を確認
- **ミッドストリーム**: 23 本の「N バイト目で切断」シナリオ

特に **SSE テストの書き方**が独特で、`AsyncIterator[bytes]` を文字列チャンクに decode してイベント列に再構成するヘルパー (`tests/helpers/sse.py`) があり、そこが `message_start → content_block_start → content_block_delta* → content_block_stop → message_delta → message_stop` の Anthropic 順序契約を明示的にアサートします。

## なぜ依存を 5 個に絞るのか

**サプライチェーン**の観点で、推移的依存が多いと:

- npm 的な typosquatting / 悪質パッケージ混入リスク
- 新規リリースのたびに全依存の diff を追う必要がある
- `uv sync` が遅くなる
- CI のイメージサイズが増える

CodeRouter の 5 個 (fastapi / uvicorn / httpx / pydantic / pyyaml) は、それぞれ広く監査されている主要 OSS。これ以上増やさないのは自己規律で、機能追加のたびに「`itsdangerous` を足せば cookie が...」「`cryptography` があれば...」という誘惑がありますが、現時点で全機能を 5 個で実現できています。

この「5 個縛り」は、初期設計の決定というより**各リリースの機能追加時の非機能要件として毎回確認されている**もので、`plan.md` と `pyproject.toml` の依存欄が短いままであることで監査されています。

## まとめ

- **Ingress と provider kind を直交化** した 2×2 マトリクスが、クライアント種別とプロバイダ種別の全組を動かす。
- **Mid-stream フォールバックガード** は、「first byte 判定」と「`event: error` 合成」の 2 点で実装される。これが無いとフランケン応答が起きる。
- **ステートフル SSE フィルタチェーン** は、部分タグ保留と EOF flush の実装が全体の複雑さを決める。OpenAI / Anthropic 経路で flush 戦略を変える必要がある。
- **Doctor の実地プローブ** は、`httpx` で 1 枚下のレイヤを観測して、自プロダクトの内部を信用しない設計。Exit code 0/1/2 のセマンティクスで CI 統合。

全部で 5 依存、453 テスト、Python 3.12+、MIT。書いてみると、「小さいのに地味に難しい」ところが目に見えやすい領域です。
