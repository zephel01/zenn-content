# LLM ルーターの『自動プロファイル選択』を rule-based でどこまでやるか

## この記事で扱うこと

[CodeRouter](https://github.com/zephel01/CodeRouter) v1.6 で入れた `auto_router` という feature の、**分類器としての設計判断**を書きます。やっていること自体は「リクエスト本文を見てプロファイル (≒ どのモデル群に流すか) を自動で選ぶ」という、書いてみれば一行で済む話です。ただこの一行を実装するときに、**「LLM に投げて分類させるべきか」「rule-based でいいのか」「rule を何種類用意すべきか」「閾値はどう決めるか」**で結構悩んだので、その内訳を残しておきます。

結論だけ先に書くと、**matcher は 4 種類、bundled rule は 2 本 + writing fallback、精度より解釈可能性を優先**というところに着地しました。

## 1. なぜ rule-based を選んだか — LLM-as-classifier は見送り

最初に検討したのは「0.5b 級の小型 LLM を常駐させて分類器にする」案です。精度は確実に rule-based より高くなります。ただし払うコストが重く:

- 常駐プロセスが 1 つ増える。運用監視対象も 1 つ増える
- 初回リクエストに +200〜500ms。Claude Code のような対話 UI では体感で分かる
- **誤判定時にストリームを巻き戻せない**。最初の 1 token が流れ始めた後では、プロファイルの切り替えはもう手遅れ
- 分類結果がログに残っても、**なぜその分類になったかを人間が読めない** (LLM の内部状態)

4 番目が一番大きくて、CodeRouter の基底哲学である「ログ = 観測の単一ソース」(v1.5 のダッシュボードもこれに立脚) と整合しません。**rule id が確定的に残って、閾値が数字で読める**状態を保つ方が、あとでダッシュボードから原因を遡れます。

というわけで v1.6 は rule-based に倒しました。v2 でプラグイン機構が入ったら、`classify` と同じ signature の実装を差し替えられるようにしてあるので、その時に LLM 分類器を外部プラグインで足す選択肢は残しています。

## 2. Precedence 挿入位置 — どこに "auto" を置くか

v0.6-D までの precedence はこうです:

```
body.profile > X-CodeRouter-Profile > X-CodeRouter-Mode > default_profile
```

v1.6 で `auto_router` を挟む場所を、**default_profile の 1 段上**に決めました:

```
body.profile
  > X-CodeRouter-Profile
  > X-CodeRouter-Mode
  > auto_router                ← default_profile == "auto" のときだけ発火
  > default_profile
```

ここには 2 つ設計判断があります。

1 つ目: **auto の発火条件は `default_profile == "auto"` という sentinel**。これで既存ユーザー (v1.5 で `default_profile: coding` を使っていた人) は **yaml に 1 文字も触らないかぎり動作変化ゼロ**。後方互換を完全に保ちます。

2 つ目: **imperative な override (body / header) は常に auto より上**。「普段は auto に任せているけれど、この 1 リクエストだけ手動指定したい」が常に可能です。上級者の per-request override path は既存経路そのままで、auto_router は declarative な層を 1 段だけ拡張しています。

`auto` を profile name 側の予約語にしたので、`profiles[]` に `name: auto` が定義されていたら起動時 fast-fail します。これは `examples/providers.auto.yaml` を見たユーザーが「auto って profile か」と思って真似しないためのガード。

## 3. Matcher を 4 種類に絞った理由

v1.6 初期で提供する matcher は次の 4 つです:

```python
class RuleMatcher(BaseModel):
    has_image: bool | None = None
    code_fence_ratio: ThresholdSpec | None = None   # {">=": 0.3}
    content_contains: str | None = None
    content_regex: str | None = None

    @model_validator(mode="after")
    def _exactly_one(self) -> Self:
        set_fields = [k for k in self.model_fields if getattr(self, k) is not None]
        if len(set_fields) != 1:
            raise ValueError(
                f"RuleMatcher must have exactly one field set, got {set_fields}"
            )
        return self
```

「`has_image` と `content_contains` を AND で組みたい」のような複合条件は v1.6 では書けません。pydantic の discriminated union として **1 rule = 1 matcher** に絞っています。理由は:

- 複合条件を yaml で書き下せるようにすると、DSL になる。「`and:` / `or:` / `not:` を並べ始めると、もう `cue` や `rego` を入れた方が健全」という世界に入る
- CodeRouter 本体の「依存 5 個」という自己規律に反する
- 複合ロジックが必要な中級者は、**順序違いの複数 rule で等価なものが書ける**ことが多い (first match wins)

逆に「この 4 種類なら、現実のリクエストの 9 割はカバーできる」ところから削りました。`has_image` / `code_fence_ratio` の 2 つが bundled で実装するに値する信号、`content_contains` / `content_regex` の 2 つがユーザー定義ルール用の逃げ道です。

**matcher を追加したくなったら**、pydantic model に optional field を足して `_exactly_one` にまかせれば、既存 yaml が壊れずに通ります (すべて None のまま)。これは後方互換性を担保するための地味な設計で、実際 alpha テスト中に `token_count_estimate` matcher を足したくなったときに yaml 側を一切触らず試せました (結局 v1.6 初期には入れませんでしたが)。

## 4. Bundled rule は 2 本 + writing fallback — 順序と閾値の根拠

zero-config で動くための bundled rule は 2 本、どれにも当たらなかった時の落ち先として `default_rule_profile: writing` を 1 つ。**順序が意味を持つ**ので並び順そのものが設計:

```python
_BUNDLED_RULES = [
    AutoRouteRule(
        id="builtin:image-attachment",
        profile="multi",
        match=RuleMatcher(has_image=True),
    ),
    AutoRouteRule(
        id="builtin:code-fence-dense",
        profile="coding",
        match=RuleMatcher(code_fence_ratio={">=": 0.3}),
    ),
]
_BUNDLED_DEFAULT_RULE_PROFILE = "writing"
```

### 画像優先の理由

画像があったら `multi` (vision 対応モデル) に振り分ける rule を最優先に置いたのは、**vision を持たないモデルに画像を渡した時の失敗が一番痛い**からです。`qwen2.5-coder:7b` は "I can't see the image" を返すだけで、モデル側にフォールバックのヒントがない。コード密度が高い画像 (スクリーンショットのコード) であっても、画像が入っていたら `multi` に倒す、という判断です。「コードと画像が両方ある」は `multi` 側に寄せてあります。

### `code_fence_ratio` の閾値 0.3 の根拠

これが一番迷った数字です。決め方は `messages[*].content` を concat → ` ```...``` ` で囲まれた部分の文字数 ÷ 全体文字数、で計算します。社内実測で以下の分布が出ました:

- **「このコード直して」系** (コード引用 + 短い質問): 0.25〜0.45
- **「この関数の使い方教えて」系** (コード引用 + 長めの解説質問): 0.15〜0.30

両者の境界を 0.3 に置くと、前者が `coding`、後者が `writing` に振り分かります。`writing` に振り分けた方は「一般的な文章生成寄りのモデル (例: `qwen2.5:7b`) が解説文を書く」扱いになるので、体感でそちらのほうが質が高い出力になりました。

閾値 0.3 は yaml で `{">=": 0.3}` と書いてあって、ユーザーが書き換え可能です。0.25 に下げれば「コード引用が少しでもあれば coding」側に倒すので、全リクエストがコード文脈である人 (開発者の typical case) は下げて使うと合います。

### fallthrough が `writing` である理由

画像もコードもない要求は **`writing` に落とす** のが default です。これは「CodeRouter を使うのは開発者だけではない」という前提で、文章生成寄りを fallback にすることで、Slack の下書きや日報といった非コード用途で変な返りをしにくくなります。コーダーモデルに日本語文章を書かせると語彙が狭くなる (コード文脈の単語に引っ張られる) 現象が観測されていたのも理由のひとつ。

## 5. User override は replace-only — prepend / append は出さなかった

ユーザーが `auto_router:` を yaml に書いた瞬間、**bundled は一切マージされず丸ごと上書き**です。

```yaml
default_profile: auto
auto_router:
  rules:
    - id: user:translate-intent
      profile: writing
      match:
        content_contains: "日本語で"
    # bundled は入らない — 上の 1 本だけが全て
  default_rule_profile: writing
```

prepend / append / replace の 3 モードを用意する案もありましたが、**yaml を読んだだけで最終の評価順が読めなくなる**のを嫌って replace-only に倒しました。bundled を踏襲しつつ拡張したい人向けに、`examples/providers.auto-custom.yaml` として bundled を yaml 化したテンプレを同梱してあり、そこから copy & edit する導線にしています。

first match wins のセマンティクスも、このテンプレの順序を見れば自然に理解できるはず、という想定です。

## 6. 失敗モードは **全部起動時 fast-fail**

auto_router の rule に関するエラーは、**リクエスト時には発生させない**設計です:

| 状況 | v1.6 の挙動 |
|---|---|
| bundled profile (multi/coding/writing) のどれかが未定義 | 起動時 fast-fail。欠けた名前を列挙して `examples/providers.auto-default.yaml` への誘導 |
| `rules[*].profile` が `profiles[]` に無い | 起動時 fast-fail (既存 `mode_aliases` validator と同じ経路) |
| 未知 matcher 型 (将来 yaml を古い CodeRouter が読む) | pydantic の `extra=forbid` で起動時 fast-fail |
| `content_regex` のコンパイルエラー | 起動時 fast-fail、yaml 上の行番号付き |
| 全 rule が non-match | `default_rule_profile` に落ち、`auto-router-fallthrough` event を 1 本吐く |

最後の 1 つだけがリクエスト時の挙動で、それも「エラー」ではなく「観測可能な default 落ち」。Prometheus counter `coderouter_auto_router_fallthrough_total` でレート監視できて、**全 rule が漏らしているパターン** (bundled rule ではカバーできない典型リクエスト) を事後発見できます。

起動時 fast-fail に寄せている理由は、ログの単一ソース哲学と同じで、**運用中に「なんかたまに変な profile に行く」が発生する曖昧さを消す**ためです。起動時に落ちていなければ、リクエスト時の挙動は rule table で完全に決定されている、という保証を付けたい。

## 7. ログとダッシュボードへの接続

分類が走るたびに 1 本、event を吐きます:

```python
logger.info(
    "auto-router-resolved",
    extra={
        "rule_id": "builtin:code-fence-dense",
        "resolved_profile": "coding",
        "signals": {
            "has_image": False,
            "code_fence_ratio": 0.42,
            "content_len": 1834,
        },
    },
)
```

v1.5 の `MetricsCollector` は `extra` を whitelist filter で ring buffer に取り込む設計になっていたので、`rule_id` / `resolved_profile` の 2 キーを whitelist に足すだけで、Recent Events パネルにそのまま現れます。**ダッシュボードの UI 側には手を入れていません**。

fallthrough は別 event (`auto-router-fallthrough`) と別 counter で出しています。統合してもいい議論はしましたが、Prometheus で「fallthrough 率が 0 か 0 でないか」を単独メトリクスで見たかったので分割しました。観測上、fallthrough が 5% 以上続けば「bundled rule に 1 本足す時期」の signal になります。

## まとめ

- LLM-as-classifier は **精度 > 解釈可能性** を許容できないので見送り。rule-based に倒した
- Precedence は `X-CodeRouter-Mode > auto_router > default_profile`。sentinel `default_profile: auto` で opt-in、**既存ユーザーは動作変化ゼロ**
- matcher は 4 種類で discriminated union。1 rule = 1 matcher の制約で DSL 化を拒否した
- bundled rule は 2 本 (`has_image` / `code_fence_ratio ≥ 0.3`) + fallthrough=`writing`。閾値 0.3 は実測分布の境界
- user override は replace-only。prepend / append は yaml の読みやすさのために見送った
- rule の妥当性は全部起動時 fast-fail、リクエスト時の曖昧さをゼロに
- ログ event 2 種 (`resolved` / `fallthrough`) で v1.5 ダッシュボードに刺さる

次に手を入れるとすれば、実装空けてある `token_count_estimate` matcher (コンテキスト長に応じた振り分け) か、プラグイン機構で LLM classifier を差し替え可能にする v2 方向です。そこに行くまでは、この 4 matcher / 3 rule で現実のリクエストの 9 割は素直に流せる、というのが v1.6 のベットです。
