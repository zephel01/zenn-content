# 同じ Ollama + qwen2.5-coder なのに、人によって動く / 動かないが分かれる理由

## TL;DR

「ローカル Ollama + Claude Code」の組み合わせで、**同じモデル・同じ Ollama バージョンを使っても、ユーザ A では快適に動き、ユーザ B では tool が全部空振りする** ——この再現性の低さには、ほぼ例外なく 6 種類の構造的原因のいずれか（または複数）があります。本稿はその 6 種類を分類し、それぞれを**ヒューリスティクスではなく実地プローブで区別する方法**を示します。具体的な検出ロジックは OSS の CodeRouter (`coderouter doctor --check-model`) に実装されている 6 プローブを題材にします。

対象読者: Claude Code / Agents をローカル LLM で動かそうとしていて「とりあえず動いたが、設定が正しいのか不安」「動かないときに何を疑えばいいか分からない」と感じている中〜上級者。

## 問題設定: 観測可能な失敗が多すぎる

Claude Code + ローカル Ollama の構成で起きる「動かない」は、表層症状だけで言っても少なくとも以下がある:

1. 最初の 1 文字が返る前にタイムアウトする
2. 最初の 1 文字は返るが、文の途中で突然切れる
3. テキストは返るが tool が一切呼ばれない
4. tool 呼び出しは来るが、JSON パースで壊れる
5. 画面に `<think>...</think>` とか `<|eot_id|>` みたいなマーカーがそのまま出る
6. 応答の品質が明らかに悪い（学習済みのはずの知識を知らないフリをする）

これらの症状と原因の対応は 1:1 ではありません。たとえば「応答品質が悪い」は、**プロンプト先頭の切り詰め**と**tool 呼び出しフォーマット違反**と**thinking 漏れ**のどれからも発生します。ユーザが「モデルがバカ」と結論して別のモデルに乗り換えた結果、同じ症状が別モデルで再現する、というのがよくある顛末です。

## 失敗モードの分類

CodeRouter の `doctor --check-model` に実装されている 6 プローブは、まさに「同じ観測症状に対して複数の原因が並立する」状態を**因果ごとに切り分ける**ために設計されています。プローブの並びはパイプラインの早い順:

```
auth  →  context-window (num_ctx canary)  →  tools  →  streaming (num_predict)  →  reasoning-leak  →  anthropic-thinking
```

以下、それぞれ何を検出しているかを見ていきます。

### 1. `auth` — そもそも鍵が通っていない

最初に疑うべき。`Authorization: Bearer $TOKEN` で `/v1/models` を叩くだけで判定できるが、問題は「Ollama のようにトークン検査しないローカルランナーにも対しても同じプローブを走らせる」こと。Ollama は**未知のヘッダを無視**するので、`Bearer dummy` でも 200 を返します。これを「auth OK」と誤判定すると、以降のプローブが全部無意味になる可能性があります。

CodeRouter はここで `kind` を見ます。`kind: openai_compat` で `base_url` が `:11434` を含む、または `extra_body.options.num_ctx` が宣言されている場合「これは Ollama 形状」と判定し、auth は素通し扱い。`kind: anthropic` の場合は `x-api-key` を必須にして、無ければ SKIP せず FAIL 扱い。

### 2. `num_ctx` プローブ — 入力側のサイレント切り詰め

最大のハマりポイント。Ollama は `options.num_ctx` を指定しないと **2048** を既定値として採用します。Claude Code の初期プロンプトは簡単に 5K〜10K トークンに達するので、デフォルトのまま使うと**プロンプト先頭がサイレントに捨てられます**。モデルは「Claude は親切なアシスタントです... (中略) ... 以下のツールが使えます: ..." の手前を全く知らない状態で、ユーザの最新発言だけを見ることになります。

**検出方法**: 先頭に一意な canary 文字列（例: `"CANARY_XRA7Q82K please echo this token verbatim"` のような 8K トークンくらいの長めのプロンプト）を仕込んで、モデルにエコーさせる。canary が応答に含まれていれば切り詰めは起きていない、含まれていなければ切り詰められている。

ここで「トークンのエコーなんて、学習データに依存して不安定では」と思うかもしれません。実際その通りで、CodeRouter のプローブは「canary 文字列を 32 文字程度の**学習データに絶対存在しない** `[a-f0-9]` のランダム列」にして、「このトークンを出力のどこかに含めて」というインストラクションを**プロンプト末尾**に置きます。末尾を指示にするのがポイントで、num_ctx で先頭が落ちてもモデルは「何を言われたか」は分かり、canary だけが（切られていれば）知らないまま応答することになります。

**4 分岐**:

| canary 観測 | `extra_body.options.num_ctx` 宣言 | 判定 |
|---|---|---|
| あり | 十分大きい (≥ 32768) | OK |
| あり | 宣言なし | OK だが宣言を推奨（将来の Ollama 既定変更に備える） |
| なし | 宣言なし | NEEDS_TUNING — `num_ctx: 32768` のパッチ提示 |
| なし | 既に宣言あり | モデル本来の context 上限が天井の可能性、モデル切り替え提案 |

この分類が大事なのは、「canary なし & 宣言なし」を「モデルが悪い」に誤帰責するのが非常によくあるからです。Ollama を入れたての人は `num_ctx` の存在自体知らないので、「qwen2.5-coder って思ったよりバカだな」で話が終わります。

### 3. `tools` プローブ — tool_use ケイパビリティの実地確認

`capabilities.tools: true` を YAML で宣言していても、モデルが実際に tool を呼べるとは限りません。特にローカル小型モデルは、プロンプトに `tools` を添えた瞬間に**フォーマット違反**で失敗する（JSON を間違った階層で出す、引数名をプロンプトの和訳にする、生テキストで吐く、etc.）。

**検出方法**: `get_time` のような 1 引数 tool を宣言して、「今の時刻を教えて」と投げる。応答が `tool_calls: [{function: {name: "get_time", arguments: "{}"}}]` 形式で正しく返るか確認。

NG パターンは豊富で:

- `choices[0].message.content` に `"get_time()"` とテキストで来る → balanced-brace リペアの対象
- `tool_calls` が空で `content` に自然言語の時刻が来る → モデルがそもそも tool 概念を持っていない、`capabilities.tools: false` 推奨
- `tool_calls` は来るが `function.arguments` が `null` → Anthropic 形式だと `tool_use.input: {}` に対応。CodeRouter はこれを許容

CodeRouter のリペア層（v0.3-A/D）は、この NG を**非ストリーミングとストリーミング両経路で自動修復**します。直接接続ではこのレイヤがないので、モデルが「tool を呼ぼうとしたがフォーマットを間違えた」ケースが全滅に見えます。

### 4. `streaming` プローブ — 出力側の早期終了

こちらは `num_ctx` と対になる出力側の罠。Ollama の `options.num_predict` 既定値は古いバージョンで **128**、新しいものでも短めです。tool 呼び出しを含むやや長い応答は簡単に 128 トークンを超えるので、**応答がフォーマット途中で `finish_reason: length` で切られます**。

ユーザには「文の途中で黙った」「JSON の `}` が来る前に切れた」と見えます。後者は特に厄介で、CodeRouter のリペア層は不完全な JSON を修復しないので、tool 呼び出しが完全に脱落します。

**検出方法**: 決定的な長い応答を要求（「1 から 30 まで数えてください、各数字を 1 行ずつ」）し、SSE ストリームを消費して `finish_reason` と最終コンテンツ長を見る。

4 分岐:

| finish_reason | 数字個数 | 判定 |
|---|---|---|
| `stop` | 30 個ある | OK |
| `length` | 30 個未満 | NEEDS_TUNING — `num_predict: 4096` のパッチ提示 |
| `stop` | 30 個未満 | モデルが指示に従えていない（学習不足） |
| ストリームチャンクがゼロなのに 200 OK | — | 上流が `stream: true` を無視している or 非標準フレーミング。サーバー修正案件 |

副次的に、SSE フレーミングが壊れている上流（vLLM の古いバージョン、一部 OpenAI 互換フォーク）もこのプローブで副産物として検出できます。

### 5. `reasoning-leak` プローブ — 非標準フィールドと `<think>` 漏れ

最近のモデルは「推論過程」を出力に混入してきます。混入パターンは主に 3 種類:

**A. `<think>...</think>` をコンテンツに埋め込む**

DeepSeek-R1 蒸留系、QwQ 32B、Gemma 4 の thinking モード、一部 Qwen3 系。`choices[0].message.content` に `"<think>ここで考察...</think>実際の応答"` のように埋め込まれてくる。

**B. `message.reasoning` / `delta.reasoning` という非標準フィールドで返す**

OpenRouter 経由の `openai/gpt-oss-120b:free` で確認。OpenAI 公式 Chat Completions スキーマには存在しないフィールドで、厳格クライアント（Pydantic で strict=True にしているような）は 422 を返します。

**C. `<|im_end|>`, `<|eot_id|>`, `<|turn|>`, `<|channel>thought`, `<|python_tag|>` などの stop マーカーが残る**

トークナイザの停止トークンがデコーダで文字列化されて漏れるパターン。Llama 3 系、Qwen 系、gpt-oss などで見る。

**検出方法**: `httpx` を直叩き（アダプタの strip レイヤを経由せず）して、生レスポンスボディを観測。`<think>` や `<|eot_id|>` や非標準キーを正規表現で探す。見つかったら:

- A → `output_filters: [strip_thinking]` パッチ
- B → `output_filters: [strip_reasoning_field]` or `capabilities.reasoning_passthrough: false` パッチ
- C → `output_filters: [strip_stop_markers]` パッチ

CodeRouter のフィルタチェーン (v1.0-A) は**ステートフルなストリーミング対応**で、SSE チャンクをまたいで `<thi` / `nk>` に分割された部分タグもちゃんと拾います。素の sed 的な正規表現では捕まらないので、自前で書くなら相応に頑張る必要があります。

### 6. `anthropic-thinking` プローブ — Anthropic 系プロバイダの body フィールド受理確認

Claude 3.7 / 4 / 4.5 / Opus 4.6 は `thinking: {type: "enabled", budget_tokens: ...}` というフィールドをリクエストボディに乗せられます。これは Claude Code が **adaptive thinking** で使ってくる機能ですが、全ての Claude モデルが受けつけるわけではありません。古い `claude-3-haiku-20240307` に `thinking` を付けると 400 で弾かれます。

CodeRouter は `kind: anthropic` プロバイダに対して、Anthropic ネイティブ形状で `thinking` ボディ付き small リクエストを送り、200 / 400 を見て capability を判定。400 なら `capabilities.thinking: false` を `providers.yaml` にパッチ提案します。さらに capability レジストリ (v0.7-A `model-capabilities.yaml`) と突き合わせ、ユーザの宣言が登録済みファミリ推定と食い違えば警告します。

## なぜ「人による再現性」が低いのか

ここまで 6 プローブを見ると、**「動いた / 動かない」の分岐点が Ollama 側の設定・Claude Code のバージョン・モデルの学習ロット・OS の環境変数のどこでも発生し得る**ことが分かります:

- Ollama を最近入れた人は `num_ctx` の既定が違うバージョンを使っているかもしれない
- 別チャンネル（Docker や ollama-webui 同梱版）で入れた人は `num_predict` が上書きされている可能性
- `qwen2.5-coder:7b` の GGUF を自分でビルドした人は学習チェックポイントが微妙に違う可能性
- Claude Code が途中でシステムプロンプト仕様を変えた（→ 必要 `num_ctx` 閾値が変動）
- ユーザの環境変数 `OLLAMA_NUM_CTX` が全モデル既定を変えている

これが「ブログ記事通りに設定したのに動かない」の原因の 8 割です。**設定を真似るのではなく、自分の環境で実地プローブして原因を切り分ける**のが唯一の安定解です。CodeRouter の doctor は「プローブ → 判定 → YAML パッチ提示」を 5 秒で回してくれるので、ブログの設定を貼る代わりに doctor の出力を貼る方が有益です。

## プローブの戦略的配置順

doctor の 6 プローブの順番には意図があります。同じ観測症状（例: 応答が短い）を生む原因が複数あるとき、**より根本的な原因から判定する**ことで、後段プローブのレポートを誤読しないで済みます。

```
auth      → ここで落ちれば他は SKIP
num_ctx   → 入力切り詰めは出力の全症状に波及するので早い段階で
tools     → tool フォーマット違反は reasoning 漏れの誤帰責を招くので reasoning より前
streaming → num_predict の出力切り詰めは tool の後でないと「tool 不対応」と誤判定する
reasoning → ここまで来れば出力は十分長く、reasoning 漏れが観測できる
thinking  → Anthropic 専用。最後でよい
```

この順序を守らないと、たとえば「tools プローブが NG」の原因が実は「プロンプトが num_ctx で先頭切れてて tools 宣言が読めてなかった」だった、みたいな誤帰責が簡単に起きます。実際、v0.7-B 以前のヒューリスティクス（プローブではなくログベースの推論）は「tools NG = モデルに tool_use capability がない」と判定していて、誤パッチ（`capabilities.tools: false`）を量産していました。v1.0-B の canary プローブ挿入で「num_ctx が真の原因なのに tools に帰責する」バグが消えた、というのが過去の retrospective に記録されています。

## doctor の "NEEDS_TUNING" と "UNREACHABLE" の分け方

パッチ提示型の診断ツールは、**「直せるもの」と「直せないもの」を終了コードで区別する**と運用がしやすくなります。CodeRouter では:

- `0` — 全プローブ OK、宣言と実挙動が一致
- `2` — NEEDS_TUNING: 実挙動が宣言と食い違うが、`providers.yaml` の変更で解消可能。パッチを標準出力に
- `1` — UNREACHABLE: 復旧不能（auth 失敗 / 到達不能 / 未知プロバイダ名）。手動介入必須

CI で `coderouter doctor` を流すなら、`|| true` せずに exit code で分岐する使い方ができます:

```bash
coderouter doctor --check-model local-ollama
case $? in
  0) echo "OK" ;;
  2) echo "Patch suggested above; apply and re-run"; exit 0 ;;
  1) echo "Escalate: unreachable"; exit 1 ;;
esac
```

## 実装レベルの補足: `httpx` 直叩きの必然性

reasoning-leak プローブだけは CodeRouter の通常アダプタを経由しません。これは、通常経路で呼ぶと**アダプタの strip レイヤがすでに `<think>` を剥がした後のボディ**が観測されてしまい、「漏れは無い」と誤判定するため。プローブは `httpx` で直接上流を叩き、生ボディで判定します。

逆に言うと、doctor でプローブを自作する場合は「検出したい現象の直前のレイヤまで**自前で再実装する**」必要があります。アダプタ経由で reasoning 漏れは測れないし、CodeRouter の SSE パーサを通してしまうと Anthropic 系の `content_block_delta` が OpenAI 形式にすり替わっていて streaming プローブが無意味になります。Doctor は「**自分のプロダクト内部を信用せず、httpx で 1 枚下のレイヤを観測する**」という一貫した設計になっています。

## まとめ

- 「ローカル LLM + Claude Code が動かない」の原因は 6 系統。**観測症状と原因は 1:1 ではない**ので、症状だけで原因を特定するのは無理。
- 6 プローブを**意図した順番**で回すことで、より根本的な原因から判定できる。
- プローブは**ヒューリスティクス（ログの text match 等）ではなく実地プローブ**で、canary / 決定的入出力 / 生ボディ観測を併用する。
- 結果は "OK / NEEDS_TUNING + YAML パッチ / UNREACHABLE" の 3 値に落とし込むと、CI にも人にも扱いやすい。

ブログの設定をコピペする運用から卒業するには、**自分の環境で doctor を回す**。「人による再現性が低い」のは、**プローブしていないから**です。
