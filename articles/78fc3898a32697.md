## TL;DR

- 2026年4月22日、Cloudflare公式ブログで **「Agents + Stripe Projects」** 統合が発表
- そして **本日4月30日、ついに本番提供が開始**
- AIエージェント（Cursor / Claude / Windsurf など）が、人間の手を介さずに **Cloudflareアカウント新規作成 → 有料プラン契約 → ドメイン購入 → APIトークン取得 → 本番デプロイ** まで一気通貫で実行できるようになった
- 人間がやるのは **Stripe CLIで数コマンド叩く + OAuthで一度だけ承認** だけ
- さらに、Stripe Atlasで法人化した新規スタートアップには **Cloudflareクレジット$100,000** が付与されるおまけ付き

僕もStripeアカウントを持っていたので、本番提供開始当日に試してみました。素直に「未来きた」という感想です。本記事では、何が新しいのか、どう使うのか、運用上の注意点や落とし穴まで、できるだけ具体的にまとめます。

## なぜ今これが画期的なのか

過去1年、AIコーディングエージェントの賢さは「コードを書く」段階ではすでに人間並みに到達しつつありました。しかし、実際にプロダクトを世に出すまでの**最後の1マイル**には、ずっと壁が残っていました。

その壁とは、**「クラウドアカウントの開設」「課金情報の登録」「APIキーの発行と保管」「ドメインの購入」「本番デプロイ」** という、いわゆる "DevOps" の入り口部分です。コードはエージェントが書けても、これらは結局人間がブラウザを開き、フォームを埋め、コピペで橋渡しをしないと先に進みませんでした。

今回の Stripe Projects × Cloudflare 統合は、その壁を **API化された「業務手順」のレイヤー** に置き換えたものです。エージェントは Stripe Projects というインターフェースを通して、「アカウントを作って」「サブスクを開始して」「ドメインを買って」「APIキーをくれ」と一連のリクエストを発行し、Stripe が **支払い・KYC・トークン発行** を仲介する。Cloudflare はそのリクエストに応じてリソースをプロビジョニングする。

要するに、**「人間がブラウザで踏んでいた手順」が、エージェントが叩ける CLI とプロトコルになった** ということです。これがいちばん大きな変化だと思います。

## 何が変わったのか（Before / After）

これまでは、AIエージェントに「Webアプリを作ってデプロイしておいて」と頼んでも、どこかで必ず人間の作業が残っていました。Cloudflareアカウントの作成、クレジットカード情報の入力、APIトークンのコピペ、ドメインの購入──そのどれもが「エージェントの自動化フロー」を分断する原因でした。

今回の統合で、その分断がほぼなくなります。

| 項目 | 従来 | Stripe Projects導入後 |
| --- | --- | --- |
| アカウント作成 | 人間が手動 | エージェントが自動（OAuthで承認のみ） |
| 支払い情報入力 | 人間が手動 | Stripeがトークン化して安全に処理 |
| APIトークン発行 | 手動でコピペ | 自動取得・自動保存（vaultに格納） |
| ドメイン購入 | 人間が手動 | Cloudflare Registrarで自動購入 |
| サービス追加 | 人間がダッシュボード操作 | `stripe projects add` 一発 |
| 所要時間 | 10〜30分 | 数分（OAuth承認1回のみ） |
| 環境変数の引き継ぎ | `.env`を都度管理 | `.projects/`が自動で同期 |

特筆すべきは、**カード番号やAPIキーがエージェントに直接渡らない** という設計です。Stripeが支払いを仲介し、Cloudflareへのアクセスは一時的なOAuthトークンで行われるため、エージェントに過剰な権限を渡さずに自動化できます。

## Stripe Projectsの仕組みをざっくり理解する

このまま手順に進んでもいいのですが、仕組みを掴んでおくとトラブル時に強いので、簡単に整理しておきます。

### 登場人物

- **Stripe**: 支払い・課金・KYC のハブ。プロバイダーと契約するときの「クレカ的な存在」。
- **プロバイダー**: 実際にサービスを提供する事業者。Cloudflare のほかに Vercel、Clerk、PostHog、Neon などが対応。
- **AIエージェント**: Cursor / Claude Code / Windsurf など。Stripe Projects のスキル定義を読み込み、CLIコマンドを叩く。
- **あなた（人間）**: 全体の意思決定者。承認ダイアログでGo/NoGoを判断する。

### Projectの中で起きていること

1. `stripe projects init` でプロジェクトを切ると、ローカルに `.projects/` フォルダと **エージェント向けスキルファイル** が生成されます。
2. エージェントは `stripe projects catalog` でカタログ（提供サービス一覧）を取得。
3. `stripe projects add <provider>/<service>` を叩くと、Stripeが間に入ってプロバイダーのアカウント作成・サブスク開始を行い、APIキーを発行。
4. キーは `.projects/vault/` 配下に暗号化保存。実行時には環境変数として注入されます。

つまり「**プロジェクトという単位で複数プロバイダーをまとめて契約・管理する**」という仕組みになっており、プロジェクトを削除すれば紐づくサブスクもまとめて停止できます。

## 必要なもの

- Stripeアカウント（支払い方法が登録済み）
- Stripe CLI（最新版）
- AIコーディングエージェント
  - 現時点で動作確認しやすいのは **Cursor、Claude Code、Windsurf** あたり
  - エージェントが `.projects/` 配下のスキルファイルを読み込めるものなら基本的に動作

## セットアップ手順

### 1. Projectsプラグインをインストール

```bash
stripe plugin install projects
```

このコマンドで、`stripe` 本体に `projects` サブコマンドが生えます。インストール後、

```bash
stripe projects --help
```

が通れば成功です。プラグイン方式なので、本体を更新しなくても Projects だけが自動で最新版になっていく設計です。

### 2. Stripeにログイン

```bash
stripe login
```

ブラウザが開いて Stripe のダッシュボードに飛ぶので、ワンクリックで認証してください。CLI に Stripe アカウントの権限が紐づきます。すでに別プロジェクトでログイン済みなら、このステップはスキップ可能です。

### 3. プロジェクトを初期化

```bash
stripe projects init my-app
```

**ここがいちばん重要なステップ**です。実行すると、カレントディレクトリに以下のような構造が生成されます。

```
my-app/
├── .projects/
│   ├── state.json        # プロビジョニング済みサービスの定義（コミットOK）
│   ├── vault/            # 暗号化された認証情報（gitignore必須）
│   └── skills/           # AIエージェント向けの手順書 / プロンプト
└── ...
```

注目すべきは `skills/` で、ここには **「エージェントが Stripe Projects を扱うときの正しい手順書」** が書かれた Markdown ファイルが入ります。Cursor や Claude Code はこのファイルを文脈として読み込み、`stripe projects` コマンドの正しい呼び出し方を理解します。

つまり、init コマンドは **「エージェントに対する取扱説明書をプロジェクトに同梱する」** 操作と捉えると分かりやすいです。

### 4. 利用可能なサービスを確認（任意）

```bash
stripe projects catalog
```

対応プロバイダーとサービスの一覧が出ます。代表的なものは次の通り（2026年4月時点）。

- **Cloudflare**: hosting / registrar / r2 / d1 / kv など
- **Vercel**: project / domain
- **Clerk**: auth
- **Neon**: postgres
- **PostHog**: analytics
- **Resend**: email
- ほか、随時追加中

エージェントに「使えるサービス全部見せて」と頼めば、このコマンドの結果を整形して返してくれます。

### 5. Cloudflareサービスを追加

```bash
# ホスティング（Workers / Pages）
stripe projects add cloudflare/hosting

# 独自ドメインも欲しい場合
stripe projects add cloudflare/registrar:domain
```

挙動は次のようになります。

- Stripeのログインメールに **Cloudflareアカウントが既に存在する場合**: OAuth画面が開き、エージェントへのアクセス許可をワンクリックで付与。
- **存在しない場合**: Stripe側が Cloudflare アカウントを自動的にプロビジョニングし、メールアドレスを引き継いだ状態で利用可能になる。

ドメイン購入の場合は、エージェントが希望のドメイン名を CLI に渡すと、空き状況確認 → 価格表示 → 購入確認 の順で進みます。**ここは購入確定の前に必ず人間に承認を求める** 設計なので、知らない間にドメインが買われている、ということは起きません。

### 6. 支払い方法の登録（必要なら）

支払い方法をまだ登録していない場合は、

```bash
stripe projects billing add
```

を実行すると、ブラウザで Stripe の支払い情報入力フォームが開きます。**カード情報は CLI を経由しません**。あくまで Stripe のセキュアなページで入力する形なので、エージェントやログにカード番号が残る心配はありません。

## エージェントへの指示例

ここまで終われば、あとはエージェントに自然言語で指示するだけです。Cursor / Claude Code / Windsurf などにそのまま貼り付けて使える例をいくつか紹介します。

### 例1: ゼロからNext.jsアプリを公開

> 「Stripe Projectsを使って、新しいCloudflareプロジェクトをセットアップしてください。シンプルなNext.jsアプリを用意して、独自ドメイン（候補は私に提案してください）を購入し、本番にデプロイするところまで自動でお願いします。」

エージェントは内部で順に：

1. `stripe projects catalog` で対応サービスを確認
2. `stripe projects add cloudflare/hosting` でホスティング契約
3. ドメイン候補を3〜5案提示し、ユーザーが選択
4. `stripe projects add cloudflare/registrar:domain` でドメイン購入
5. Next.jsプロジェクトを `create-next-app` で生成
6. Wranglerでビルド & デプロイ
7. DNSをCloudflareの当該ドメインに紐付け

を実行します。各ステップで課金や購入が発生するときは、Stripe側からユーザーに承認を求めるフローが入るので、暴走の心配はありません。

### 例2: 既存リポジトリにホスティングだけ追加

> 「このリポジトリにCloudflare Workersのホスティングを追加して、`main` ブランチへのpushで自動デプロイされるよう設定してください。ドメインは既存の `myapp.dev` をそのまま使います。」

この場合、エージェントは `stripe projects add cloudflare/hosting` のみ実行し、リポジトリに `wrangler.toml` を生成、GitHub Actions のワークフローを追加してくれます。

### 例3: フルスタック構成

> 「Cloudflareでホスティング、Neonで Postgres、Clerkで認証、Resendでメール送信ができるフルスタック環境を作って、全部Stripe Projectsで管理してください。」

エージェントは以下のように複数コマンドをまとめて実行：

```bash
stripe projects add cloudflare/hosting
stripe projects add neon/postgres
stripe projects add clerk/auth
stripe projects add resend/email
```

それぞれのAPIキーが `.projects/vault/` に格納され、ローカル開発時には `stripe projects run` 経由で環境変数として注入されます。

## 対応プロバイダー（Cloudflare以外）

Stripe Projects はマルチプロバイダーが前提のプロトコルです。Cloudflareはあくまで「最初の大型対応」であり、他にも次のようなサービスが連携可能です。

- **ホスティング**: Cloudflare Workers/Pages、Vercel
- **データベース**: Neon (Postgres)、Cloudflare D1
- **認証**: Clerk
- **オブジェクトストレージ**: Cloudflare R2
- **解析**: PostHog
- **メール送信**: Resend

詳細は [Stripe Projects Providers 一覧](https://projects.dev/providers) を参照してください。プロバイダーは今後も追加されていきます。

## 注意点・運用Tips

実運用で気になりそうなポイントをまとめておきます。

### 支出上限のデフォルトは$100/月

デフォルトでは **1プロバイダーあたり $100/月** に制限されます。Workers の従量課金が想定外に膨らむのを防ぐ意味でも、最初はこのデフォルトのまま運用するのが無難です。引き上げたい場合は、

```bash
stripe projects billing
```

から対話的に変更できます。

### クレデンシャルとGit管理

- `.projects/vault/` と `.env` 系は **必ず `.gitignore` に追加**。
- `.projects/state.json` はリソース定義なのでコミットしてOK。むしろチームで共有するほうが、誰でも `stripe projects sync` で同じ環境を再現できます。

```gitignore
# .gitignore
.projects/vault/
.env
.env.local
```

### CI/CDでの非対話モード

GitHub Actionsなどで動かすときは、対話プロンプトを抑制するフラグを使います。

```bash
stripe projects sync --no-interactive --auto-confirm
```

CI用のサービスアカウントを Stripe で発行しておき、`STRIPE_API_KEY` を Secret に格納するのが定石です。

### 既存のCloudflareアカウントを使いたい

Stripe のログインメールと **同じメールアドレス** で Cloudflare に既にサインインしておけば、OAuthで自動連携されます。会社用と個人用でメールが違う場合、Stripe アカウント自体を使い分けるのが安全です。

### ロールバックとリソース削除

不要になったサービスは、

```bash
stripe projects remove cloudflare/hosting
```

で解除できます。サブスクリプションも合わせて停止されるので、「気付いたら課金が走り続けていた」事故を防げます。プロジェクト全体を畳みたい場合は、

```bash
stripe projects destroy
```

で関連リソースを一括クリーンアップ（ドメインなど一部は譲渡/解約フローを別途確認）。

## ハマりどころと対処法

実際に試して、「これは引っかかる人いそう」と感じたポイントを共有します。

### 1. `stripe login` 後に projects コマンドが見つからない

プラグインのインストールが完了しても、シェルのキャッシュが古いままだと `stripe projects` が見えないことがあります。新しいシェルを開き直すか、`hash -r`（bash/zsh）で解決します。

### 2. OAuth画面が開かない / ループする

ブラウザが複数アカウントでログインしている状態だと、Cloudflareの認可画面で別アカウントが選ばれてしまうことがあります。シークレットウィンドウで一度試すのが確実です。

### 3. ドメイン購入で `Insufficient funds`

Stripe側のデフォルト支出上限（$100）を超える .com 以外のTLD（.aiや.devなど）を買おうとすると、ここに引っかかります。`stripe projects billing` で月額上限を引き上げてから再実行してください。

### 4. `wrangler deploy` が `Authentication error`

エージェントが古いAPIキーを参照していると起きがちです。`stripe projects sync` を実行すると、`.projects/vault/` の最新キーが環境変数に展開され直されます。

### 5. 同じプロバイダーで複数アカウントを使い分けたい

`stripe projects link cloudflare --account-id xxx` で別アカウントを紐付け可能です。サブスクリプションも別単位になるので、本番/検証で完全に分離できます。

## 試してみた感想

「Next.jsの簡単なアプリを作ってデプロイして」と一言投げてから、独自ドメインで本番URLが返ってくるまで **おおよそ数分**。途中、ドメイン購入とサブスクリプション開始の確認ダイアログが計2回出ただけで、それ以外は完全にエージェント任せでした。

特にインディーハッカーや個人開発者にとって、「アイデア → 公開URL」の摩擦がここまで減るのはインパクトが大きいです。**プロトタイプを試す心理的コストが、ほぼゼロになる感覚** に近いです。

これまでだと、「ちょっと試したいけど、ドメイン買ってDNS設定してデプロイ環境整えて……」と考えただけで腰が重くなり、結局ローカルで止まったまま、というアイデアが何個もありました。今後はその段差が消えるので、**「とりあえず公開してみる」が新しいデフォルト** になりそうです。

## これから開発スタイルはどう変わるか

少し未来予想を書いておくと、このアップデートの本質は単なる「自動化」ではなく、**クラウドの利用単位がプロジェクトベースに揃う** ことだと思います。

- これまで: 「アカウント中心」。Cloudflare に1つアカウントを作り、その中に複数プロジェクトをぶら下げる。
- これから: 「プロジェクト中心」。1プロジェクトごとに必要なクラウドリソースが束ねられ、不要になったらまとめて畳める。

これは特に、**短命なAIプロダクト・実験プロジェクトを大量に立ち上げる** ような開発スタイルと相性が良いです。1人で月10個プロトタイプを作って2個残す、みたいな運用が現実的になります。

また、エージェントが運用主体になることで、**「課金監視」「使ってない契約の解約」「セキュリティパッチ適用」** といった日常運用も自動化対象に入ってきます。GitHub Copilot 的な「コード補完」から、本物の "Cloud Operator" としてのAIに、一歩近づいた印象です。

## おまけ：Stripe Atlas経由なら $100,000 クレジット

今回の発表に合わせて、**Stripe Atlasで法人化した新規スタートアップに対して、Cloudflareクレジット $100,000 分が付与** されるプログラムも始まりました。これから創業する人には大きいので、該当する方はぜひチェックを。

特に、Workers AI や R2 を多用するAIプロダクトを立ち上げる予定なら、$100K あれば実質1年〜2年の本番運用費がカバーできるレベルです。

## まとめ

Cloudflare × Stripe Projects で、AIエージェントがついに「人間の代わりにクラウドアカウントを運用できる存在」になりました。今までは「指示はできるけど最後の数歩は人間がやる」だったのが、本当に **アイデアを言語化するだけで本番URLが出てくる** 体験になっています。

ポイントを最後にもう一度まとめます。

- セットアップは Stripe CLI に Projects プラグインを入れて `stripe projects init` するだけ
- エージェントへの指示は自然言語でOK。承認が必要なステップだけ人間がGoサイン
- 支出上限・クレデンシャル管理・ロールバックの仕組みがしっかりしているので、暴走リスクは低い
- 対応プロバイダーは今後も増える予定で、マルチクラウド前提の運用がエージェントベースで成立する

試した方はぜひコメントでハマりどころや感想を共有してもらえると嬉しいです。
