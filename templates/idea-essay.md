## 背景：どこで引っかかったか

<!-- できるだけ具体的に。「現場でこんなことが起きて」「このコードを見て」など -->

先月、チームで △△ のリファクタを進めていた。
コードは動いていたが、ある日こんな PR が出てきた:

```ts
// before
const value = getValueOrDefault(input, DEFAULT);

// after
const value = input ?? DEFAULT;
```

一見「読みやすくなった」で終わる変更だ。
でも、数日後に別の誰かが「`getValueOrDefault` の中で元々やっていたバリデーションが消えている」ことに気づいた。

この出来事から、「可読性」と「明示性」は必ずしも同じ方向を向いていないのでは、と思い始めた。

## 自分なりの仮説

1. 短く書けるほど、読み手は「省略されていないもの」を推測できない
2. 推測を減らすためには「型」か「名前」が手がかりになる
3. つまり、記法を短くする時は **型・名前のどちらかを同時に強くする** 必要がある

:::message
このあたりは「冗長 vs 簡潔」ではなく「暗黙 vs 明示」の話に近い、と個人的には感じている。
:::

## 反論してみる

<!-- 自分の意見に自分でツッコミを入れるセクション。
     これがあるだけで記事の誠実さがグッと上がる。 -->

- 「短いコードの方が長期的には保守しやすい」という立場もある
- 自分の経験は N=1 でしかない
- そもそもチームの習熟度によって答えは変わる

つまりこの話は「絶対こうすべき」ではなく「設計レビュー時の問いかけリスト」として使うのが良さそう。
